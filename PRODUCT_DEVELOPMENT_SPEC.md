# REF Life Cycle Platform · 产品开发规格

> 文档版本：1.0  
> 原型基线：站点版本 47  
> 原型文件：[`public/demo.html`](public/demo.html)  
> 适用对象：产品经理、交互设计、前端、后端、测试、数据与集成团队  
> 文档状态：可进入需求评审与技术方案设计

## 1. 产品概述

REF Life Cycle Platform 是企业房地产与设施全生命周期管理平台。平台以租赁项目为主线，连接空间规划、租赁交易、项目交付、设施运营和成本管理，解决跨团队信息分散、流程断点、重复录入、决策数据无法追溯和合同结果偏离决策基线等问题。

### 1.1 产品目标

- 用统一项目对象贯穿需求、寻源、评估、决策、立项、合同、付款、交付、运营与退出。
- 上游已确认字段自动传递到下游，并明确显示来源、版本和同步时间。
- 同一楼宇可管理多份租赁合同，同一合同可覆盖多个楼层。
- 合同最终数据与选址评估、决策和立项基线自动比对，偏差需醒目预警。
- 待办、审批、风险、到期事项和组合分析统一呈现。
- 支持中英文与桌面、平板、移动端响应式访问。

### 1.2 成功指标

| 指标 | 建议目标 |
| --- | --- |
| 上游字段自动带入率 | ≥ 90% |
| 下游重复录入字段减少 | ≥ 70% |
| 选址至合同关键数据可追溯率 | 100% |
| 合同偏离决策基线识别率 | 100%（纳入比较清单的字段） |
| 租赁关键节点按时完成率 | ≥ 90% |
| 180 天内到期合同覆盖提醒率 | 100% |

## 2. 用户、角色与权限

| 角色 | 核心任务 | 建议权限 |
| --- | --- | --- |
| 规划经理 / 房产 BP | 空间需求、容量与选址规划 | 创建需求，编辑规划评估，读取关联项目 |
| 租赁经理 | 寻源、评估、决策、立项、合同 | 租赁域全量编辑、提交与流程推进 |
| 设计经理 | 设计检查、方案与技术风险 | 编辑设计评估，上传设计及尽调文件 |
| 项目经理 | 设计深化、采购、施工、验收与移交 | 编辑交付域，读取租赁与成本基线 |
| 设施经理 | 运营需求、工单、资产、供应商与合规 | 编辑运营域，接收项目移交 |
| 成本经理 | 预算、承诺、实际、EAC 与偏差 | 编辑成本域，复核合同成本偏差 |
| 法务 / 合规 | 楼宇合规、商务条款与合同审查 | 编辑合规及条款结论，读取决策材料 |
| 管理者 | 组合绩效、风险、成本与到期结构 | 全局只读、导出、看板配置 |
| 系统管理员 | 字典、权限、流程与集成配置 | 管理配置，不默认读取敏感业务数据 |

权限采用 `RBAC + 数据范围 + 字段级权限`：

- 功能权限控制页面、按钮和流程操作。
- 数据范围按区域、城市、部门、项目成员和管理范围控制。
- 租金、成本、保证金、供应商与合同条款支持字段级脱敏。
- 所有写操作必须记录操作人、时间、来源、前后值和 trace id。

## 4. 租赁项目类型与生命周期

### 4.1 新租 / 扩租

`需求确认 → 场地寻源 → 选址评估 → 选址确认 → 业务场地确认 → 场地信息登记 → 租赁立项 → CS 邮件同步 → 合同签署 → 拿场地 → Target Day 1`

### 4.2 续租

沿用现有场地和合同信息，进入续租评估、推荐与审批结论。审批选择：

- 继续续租：进入续租立项、合同签署与台账更新。
- 转为换租：从同一续租需求派生一个新租项目和一个退租项目。

### 4.3 换租

换租不建立第三种项目流程。系统创建两个独立项目：

1. 新租项目：负责新址寻源、评估、签约和交付。
2. 退租项目：负责原址通知、复原、结算和交还。

两个项目通过 `relation_group_id` 双向关联，并共享原需求来源，但分别维护字段、计划、状态、审批和合同。

### 4.4 退租

`退租需求 → 退租范围确认 → 业务场地确认 → 退租审批 → 正式通知 → 复原与交还 → 费用结算 → 台账关闭`

## 5. 租赁项目详情

### 5.1 顶部信息

- 项目编号、名称、类型、城市、当前阶段、健康度、负责人。
- AI 项目摘要、风险提示和下一步建议。
- 项目状态由关键流程节点自动驱动，不允许仅靠前端改状态。

### 5.2 项目团队

- 支持规划、租赁、项目、设计、机电、成本、法务、消防、行政、物理安全等角色。
- 支持新增、删除成员和一键创建项目群。
- 成员变更记录审计日志。

### 5.3 WBS 计划

- 字段：WBS 编号、节点、基线日期、实际日期、状态、负责人、说明。
- 支持新增节点、编辑和保存。
- 关键节点变更保留历史版本。
- 新租示例中选址确认子节点为 3.1–3.4，合同签署为 4.0。

## 6. 场地寻源与短名单

### 6.1 长名单 / 短名单

- 支持导入场地清单。
- 导入识别结果必须经人工确认后写入。
- 场地字段包括楼宇、楼层单元、位置、面积、预计工位、租金、物业费、交付状态等。
- 支持加入短名单、移出短名单和保存筛选总结。

### 6.2 短名单综合评估矩阵

- 楼宇、专业维度、综合得分和推荐等级在同一视图横向比较。
- 推荐意向场地与备用场地各最多一个。
- 标记必须来自当前短名单，并同步到各专业评估指标组。
- AI 可基于矩阵和专业结论生成“短进一意向场地推荐总结”，租赁经理可编辑。
- 英文操作文案为 `Generate Strategy Paper`。

## 7. 选址评估

### 7.1 楼宇合规评估

楼宇合规评估位于短名单评估中，不在推荐场地决策材料中重复出现。原“合规与技术评估”删除，技术风险移动到设计评估。

建议字段包括消防、产权、用途、出租权利、抵押查封、签约主体与第三方合规文件等。

### 7.2 成本评估

成本评估必须按以下顺序展示 14 个字段：

1. 表面租金
2. 有效租金
3. 物业费单价
4. 费用补充说明
5. 总租金
6. 总物业费
7. 押金总额
8. 押金说明（押金标准，是否可保函）
9. 复原费
10. 其他
11. 预估装修单方造价
12. 预估项目总成本
13. 施工是否指定供应商
14. 全周期单工位成本

### 7.3 设计评估

检查项顺序及标准：

| 顺序 | 检查项 | 标准 / 说明 |
| --- | --- | --- |
| 1 | 容纳人数上限（Occupancy Load） | 与设计工位数或设计使用人数比较 |
| 2 | 工位区密度 | sqm/ws；A：5.5，B：4.4 |
| 3 | 家具尺寸与间距 | 工位 1.4m × 0.7m 升降桌，间距 2m |
| 4 | 层高 | 办公区优先裸顶 ≥3.6m；会议室优先吊顶 ≥2.7m |
| 5 | 采光率 | A：70% 工位在采光面 7.5m 内；B：50% |
| 6 | 会议空间配比 | 1:8～10 |
| 7 | 卫生间配比 | 参考当地标准；建议男厕 20:1、女厕 16:1 |
| 8 | 餐饮服务配套 | 说明配餐模式、餐位数等 |
| 9 | 电梯运力 | 5 分钟载客率 >8%，单梯承载人数 ≤400 人 |
| 10 | 新风量 | 开放办公区和会议室均 ≥30 cmh/人 |
| 11 | 灯光照度 | 办公区桌面 400～500 lux，4000K |
| 12 | 供电情况 | 优先市政双回路；单方供电 ≥60W/sqm |
| 13 | 空调负荷 | 说明供冷、供热情况 |
| 14 | 设计方案 | 提供设计方案文档链接 |
| 15 | 技术风险（如有） | 填写结论和第三方技术尽调文件 |

第 12–15 项位于设计评估清单末尾。各检查项可显示小字标准，标题旁可放参考文档链接。

### 7.4 租赁条款评估

- 标题旁显示小字文档链接“租赁条款管控标准”。
- 检查提前退租、长短期续租、扩租、转租、转让、双方违约责任和复原条件。
- 结论、重要意见、附件与负责人均需保存版本。

## 8. 选址确认

选址确认仅保留五个串联子流程：

| 步骤 | 子流程 | 解锁条件 |
| --- | --- | --- |
| 1 | 上传决策文档存档 | 初始可操作 |
| 2 | 业务场地确认 | 决策文档已上传并存档 |
| 3 | 场地信息登记 | 业务场地确认已提交 |
| 4 | 租赁立项 | 场地信息登记已完成 |
| 5 | CS 邮件同步 | 租赁立项已完成 |

交互要求：

- 删除“推荐场地决策材料”板块。
- 第一步不再是保存决策结论，而是上传决策文档存档，并预留文件上传操作。
- 业务场地确认使用独立表单，并显示字段来自需求申请或决策会。
- “场地信息登记”展开工作区必须紧跟第 3 步下方，结束后才显示第 4 步。
- 场地登记完成后回填工区、楼宇、楼层、面积、起止日期和租期。
- CS 邮件同步复用已登记信息，并在发送前校验收件人、标题、字段和附件。
- 前后端均需校验流程顺序，禁止通过接口跳步。

## 9. 租赁立项

- 列表展示项目、立项编号、预算、审批状态和操作。
- 详情字段优先继承推荐场地、租赁决策、成本测算和项目团队。
- 立项阶段补充预算归属、费用明细、附件及风险审批。
- 立项完成后解锁 CS 邮件同步和合同签署准备。

## 10. 租赁合同管理

### 10.1 页面结构

页面自上而下：

1. 按楼宇查看租赁合同。
2. 当前楼宇内切换“按合同查看 / 按楼层查看”。
3. 下方保留全量“租赁合同台账”表格。

不展示右侧“合同系统联动”纯说明板块。

### 10.2 按楼宇查看

- 一级菜单为楼宇。
- 选中楼宇后展示地址、合同数量和总租赁面积。
- 同一楼宇允许多份合同，例如首租 L2–5、后续扩租 L6–10。
- 每份合同独立维护编号、类型、状态、楼层、面积、期限、提醒和商务内容。

### 10.3 按合同查看

合同卡片展示：

- 合同名称与编号
- 合同状态
- 租赁楼层
- 租赁面积
- 合同类型
- 租赁期限
- 关键提醒
- 查看合同详情

### 10.4 按楼层查看

必须使用列表 / 表格形式，不使用卡片模块。每层一行，字段包括：

- 楼层
- 对应合同
- 合同类型
- 租赁面积（分摊）
- 租赁期限
- 合同状态
- 操作

点击“查看合同详情”进入该楼层关联合同。

### 10.5 全量合同台账

表格字段：

- 合同编号
- 项目 / 场地
- 合同类型
- 租赁楼层
- 租期
- 公司合同状态
- 关键提醒
- 操作

## 11. 合同详情与决策基线偏差预警

### 11.1 字段分组

1. 合同基本信息
2. 合同金额
3. 商务条款 by 合同
4. 保证金

字段支持分组编辑、保存、来源显示和原值保留。

### 11.2 决策基线

合同中的继承字段应保存其上游基线快照：

- 来源对象：选址评估、选址决策、业务场地确认或租赁立项。
- 基线内容：字段编码、显示值、标准化值、单位、来源对象 ID、来源版本、同步时间。
- 合同阶段修改不覆盖上游基线。

建议首批比较字段：

- 租赁面积
- 起租日期、到期日期、租期
- 表面租金、有效租金、物业费单价
- 总租金、总物业费
- 保证金总额及类型
- 合同单工位成本
- 提前退租、续租、扩租、转租、转让和复原条件

### 11.3 偏差识别规则

- 文本：去除首尾空格并标准化空白后比较。
- 金额 / 单价：统一币种、单位和小数精度后比较。
- 日期：统一转换为 ISO 日期比较。
- 枚举：按枚举 code 比较，不按显示文案比较。
- 面积：统一为平方米比较。
- 复杂条款：比较结构化选项，并保留文本差异。
- 允许为字段配置容差，例如金额差异 ≤1% 仅提示、不阻断。

### 11.4 预警交互

当合同值与决策基线不一致时：

- 字段容器、字段名和当前合同值高亮。
- 字段旁显示“原决策数据 + 来源”。
- 详情顶部汇总差异字段数量。
- 编辑过程中实时比对，保存后再次由后端校验。
- 支持点击查看上游决策版本和变更记录。
- 高风险字段可配置为合同提交或签署前必须确认。

示例：

| 字段 | 原决策数据 | 合同数据 | 展示 |
| --- | --- | --- | --- |
| 合同单工位成本 | ¥6,820 / 工位 / 月 | ¥7,000 / 工位 / 月 | 红色高亮，并备注原决策数据 |

## 12. 租赁付款管理

- 按租赁项目和合同跟踪租金、保证金、税费及其他付款。
- 字段包括付款事项、项目、关联合同、计划日期、金额和状态。
- 付款前置条件：合同生效、付款计划符合合同、预算可用、收款信息已验证。
- 到期前提醒，逾期或金额偏差自动升级。

## 14. 状态机与关键校验

### 14.1 选址确认状态机

`DECISION_DOCUMENT_PENDING → BUSINESS_CONFIRMATION_READY → SITE_REGISTRATION_READY → INITIATION_READY → CS_SYNC_READY → COMPLETED`

每次转换必须校验：

- 当前状态
- 必填字段
- 文件或审批状态
- 操作权限
- 对象版本
- 幂等键

### 14.2 合同状态建议

`DRAFT → TERMS_NEGOTIATION → APPROVAL → SIGNING → SIGNED → ARCHIVED`

进入 `APPROVAL` 或 `SIGNING` 前：

- 重新计算合同偏差。
- 阻断未确认的高风险偏差。
- 固化合同字段快照和决策基线版本。

| GET | `/api/buildings/{id}/contracts` | 按楼宇获取合同 |
| GET | `/api/buildings/{id}/floors/contracts` | 按楼层获取合同占用 |
| GET / POST | `/api/contracts` | 合同台账 / 创建合同 |
| GET / PATCH | `/api/contracts/{id}` | 合同详情 / 更新字段 |
| POST | `/api/contracts/{id}/baseline/sync` | 从指定上游版本生成基线 |
| GET | `/api/contracts/{id}/deviations` | 获取合同偏差 |
| POST | `/api/contracts/{id}/deviations/recalculate` | 重新计算偏差 |
| POST | `/api/contracts/{id}/deviations/{deviationId}/confirm` | 确认偏差 |
| GET / POST | `/api/contracts/{id}/payments` | 付款计划 |

统一错误结构：

```json
{
  "code": "CONTRACT_BASELINE_DEVIATION",
  "message": "存在未确认的高风险合同偏差",
  "field_errors": {
    "cost_per_seat": "合同值与决策基线不一致"
  },
  "trace_id": "..."
}
```

所有写接口支持 `Idempotency-Key` 和乐观锁版本号。

## 16. 前端实现要求

- 建议按路由、领域组件、服务层、权限层和状态机拆分，不沿用单 HTML 文件结构。
- 服务端返回字段 code、value、unit、source 和 baseline；前端不通过显示文案判断业务逻辑。
- 合同偏差状态使用图标、颜色和文字共同表达。
- 可编辑字段必须有明确“编辑 / 保存 / 取消”模式。
- 表格支持服务端分页、排序、筛选和固定关键列。
- 楼宇、合同、楼层视图共享查询状态，返回详情后保留选中楼宇和视图模式。
- 中英文使用 i18n key；货币、面积和日期按 locale 格式化。
- 附件上传显示进度、失败重试、版本和病毒扫描状态。

## 17. 非功能要求

- 桌面首屏 LCP `< 2.5s`；普通列表 API P95 `< 800ms`。
- 关键写接口可用性 ≥99.9%，需具备重试与幂等保护。
- 满足 WCAG 2.1 AA：键盘操作、可见焦点、语义标签和屏幕阅读器提示。
- 敏感字段传输及存储加密，下载链接短时有效。
- 附件校验扩展名、MIME、大小、病毒扫描和访问权限。
- 所有字段、状态、审批、上传、发送、AI 建议和偏差确认均记录审计日志。
- 支持数据导出，但导出受权限、脱敏和水印策略控制。

## 19. 测试建议

- 单元测试：字段标准化、基线比较、容差、状态转换和权限判断。
- 集成测试：选址确认五步串联、基线生成、合同保存与偏差重算。
- 端到端测试：楼宇 → 合同 / 楼层 → 合同详情 → 修改字段 → 高亮预警。
- 契约测试：上游字段 code、单位、版本和来源结构。
- 回归测试：中英文、响应式、表格滚动、返回状态和多合同楼宇。

## 20. 建议实施顺序

1. 项目、楼宇、楼层、合同与字段字典基础模型。

8. 权限、审计、国际化、性能和可访问性完善。

## 21. 原型与生产实现边界

`public/demo.html` 是交互原型。当前数据、AI 回复、文件上传、邮件、审批、保存、合同偏差和第三方集成均为前端模拟。

生产实现必须：

- 将原型状态迁移到后端持久化模型。
- 由服务端执行权限、状态机、偏差计算和幂等校验。
- 将 AI 输出视为可审核草稿，不直接写入正式业务数据。
- 对决策基线和合同数据分别存储，禁止下游修改覆盖上游历史。
- 通过 API 和事件机制连接审批、合同、邮件、文档与成本系统。
